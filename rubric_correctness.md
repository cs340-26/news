# Assignment Rubric: Functional Correctness & Robustness

**Course:** CS 340-26  
**Assignment:** Code Quality — Part 1 of 3  
**Total Points:** 100  
**Evaluated in:** `code_quality_grade.md` in each project repository

---

## Overview

This assignment evaluates whether your code *works* — correctly, consistently, and safely. A program that only functions under ideal conditions is not production-ready. This rubric rewards teams that verify their work through automated tests, anticipate what can go wrong, and write defensively.

The three sub-criteria are evaluated independently and combined for a total score.

---

## Sub-Criterion 1a — Automated Testing (40 points)

Automated tests are the primary evidence that a team understands what their code is supposed to do and has *confirmed* that it does so. Tests should be committed to the repository, runnable with a single command, and targeted at the code the team actually wrote (not framework boilerplate).

**Accepted frameworks by project type:**

| Project Type | Framework |
|---|---|
| Python (Django / Flask / FastAPI) | `pytest`, `django.test.TestCase` |
| JavaScript / TypeScript (React) | Jest + `@testing-library/react` |
| Java (Spring Boot) | `@SpringBootTest`, `@WebMvcTest`, JUnit 5 |
| GDScript (Godot) | GUT (Godot Unit Testing) |
| C (ESP-IDF / embedded) | Unity C test framework |
| C# (Unity) | Unity Test Framework (Edit Mode or Play Mode) |

| Points | Criteria |
|--------|----------|
| 36–40 | Comprehensive test suite. Covers happy paths, at least two non-trivial edge cases, and at least one failure/error path. Tests are organized by module, named to describe what they verify, and run via a standard test runner. All tests pass. |
| 28–35 | Tests cover the core logic and at least one real edge case. Some paths are untested (e.g., error responses, boundary values) but the most important functions are verified and passing. |
| 18–27 | Tests exist and run, but coverage is narrow — e.g., only one module tested, tests check only the happy path, or assertions are trivially weak (`assertTrue(result != null)`). |
| 8–17 | One or two test functions exist but do not meaningfully verify correctness — e.g., a test that calls a function but makes no assertions, or a test file that is a copy of framework boilerplate. |
| 0–7 | No test files of any kind committed to the repository. |

**Common deductions:**
- Test file exists but no assertions: −15
- Tests declared in `package.json`/`pyproject.toml` but not in any test file: −10
- Tests present but none pass: cap at 15
- Tests only cover code generated by a framework, not code the team wrote: −20

## Sub-Criterion 1b — Edge Case Handling (35 points)

Code that crashes or silently misbehaves on unusual input is incomplete. This criterion rewards defensive programming: validating input before using it, handling API failures gracefully, and giving users meaningful feedback when something goes wrong.

| Points | Criteria |
|--------|----------|
| 32–35 | All identified edge cases are explicitly handled. Invalid or missing input returns a clear error response (HTTP 400, in-game message, stderr output). Unusual inputs do not crash the program. The user always knows what went wrong. |
| 25–31 | Most edge cases handled. One or two minor gaps (e.g., a missing warning log, a missing null check for an optional field) that would not affect most users under normal conditions. |
| 16–24 | Core feature works under happy-path conditions but known edge cases are unhandled — e.g., no required-field validation on a form, no guard against division by zero, no error message when an API call fails. |
| 8–15 | Multiple unhandled edge cases that affect normal use. Silent failures (no error message to user), missing required-field checks, uncaught exceptions visible in production logs. |
| 0–7 | No evidence of edge case consideration. The code crashes or produces obviously wrong output on any non-ideal input. |

**What evaluators specifically check:**
- Empty collections / zero items / null/None values
- Missing required form or API fields
- Out-of-range numeric values (negative counts, percentages > 100)
- Network or API failures (timeouts, non-200 status codes)
- Division by zero or invalid mathematical operations
- Re-entrant calls (e.g., double-submitting a form)
- For games: off-screen despawn, simultaneous collision, state machine transition to non-existent state

## Sub-Criterion 1c — Security & Reliability (25 points)

Code committed to a shared repository must not contain credentials, must be safe to deploy, and must not have obvious vulnerabilities left unaddressed.

| Points | Criteria |
|--------|----------|
| 23–25 | No hardcoded credentials anywhere in the repository. All secrets loaded from environment variables. App refuses to start if required secrets are absent (no silent fallback to a default). CORS restricted to known origins. Auth protects all sensitive endpoints. No debug flags in production entry points. |
| 18–22 | No hardcoded secrets. Minor issues only — e.g., CORS open to all origins, a missing auth check on a low-sensitivity endpoint. No critical vulnerabilities. |
| 12–17 | Secrets loaded from environment variables but with a hardcoded fallback (`os.getenv("SECRET", "hardcoded-value")`). Debug mode left on. Open CORS. No authentication on one or more sensitive endpoints. |
| 5–11 | Multiple security issues present: hardcoded credentials in source, debug mode on, no input validation, unauthenticated endpoints that should require auth. |
| 0–4 | Credentials committed to version control, or a critical vulnerability is present (interactive debugger enabled in production enabling arbitrary code execution, dangling pointer dereference, buffer overflow with no bounds check). |

**Common findings by language:**

| Language | Common Issues |
|---|---|
| Python (Flask/Django) | `debug=True` in production, `SECRET_KEY = os.getenv("X", "dev-secret")` fallback, `allow_origins=["*"]` |
| JavaScript | Hardcoded `http://localhost:PORT`, API keys committed to source files |
| C / C++ | Dangling pointer returned from function, hardcoded Wi-Fi credentials in `main.c`, duplicate symbol linker errors |
| Java | H2 console left exposed, no Spring Security, unauthenticated `@PostMapping` endpoints |
